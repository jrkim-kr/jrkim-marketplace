# portfolio — 커리어 자산 템플릿

architect-advisor portfolio에서 생성하는 산출물의 전체 템플릿과 예시. 본 스킬의 `SKILL.md`에서 분리하여 노이즈를 줄인다.

---

## STAR 케이스 스터디

면접·포트폴리오 제출용. 한 모듈 설계 경험을 Situation / Challenge / Action / Result로 압축한다.

```markdown
## [모듈명] 아키텍처 설계 케이스

### 배경 (Situation)
[프로젝트명]의 [모듈명] 개발. [비즈니스 맥락 1-2문장].

### 도전 (Challenge)
- [핵심 기술 과제 1: 예) 멱등성 보장]
- [핵심 기술 과제 2: 예) 분산 환경 데이터 일관성]
- [핵심 기술 과제 3: 예) 장애 복구 전략]

### 행동 (Action)
두 가지 아키텍처 방안(MVP vs 견고한 설계)을 비교 분석한 후,
[방안 B]를 선택. 그 이유:
1. [아키텍처적 판단 근거 1]
2. [아키텍처적 판단 근거 2]

### 성과 (Result)
- 시스템 견고성(鲁棒性/Robustness) 향상
- 운영 비용 절감
- [구체적 수치나 개선 지표가 있다면 포함]
```

작성 팁:
- Situation은 "누가·어디서·왜"를 1~2문장에 압축. 회사명/프로젝트명은 공개 가능 여부 확인 후 기재.
- Challenge는 기술 과제만. 일정·조직 이슈는 여기 넣지 않는다.
- Action의 "그 이유"는 decision 비교표에서 그대로 승계 가능.
- Result는 가능하면 수치(p99 레이턴시, 장애율, 처리량)로. 숫자가 없다면 "이전 대비 어떻게 달라졌는지" 서술형.

## 30초 면접 요약 (중한 이중언어)

STAR을 압축한 30초 엘리베이터 피치. 핵심 의사결정과 그 결과를 30초 안에 전달한다.

```markdown
#### 한국어 버전
[30초 분량의 한국어 면접 답변]

#### 中文版本
[30秒中文面试回答]
```

작성 원칙:
- 30초 ≈ 한국어 80~100자, 중국어 70~90자. 낭독해서 초시계로 확인.
- 직역 금지. 한국어와 중국어 각 언어 화자가 자연스럽게 들을 수 있도록 별도 작성한다.
- 구조: "무엇을 만들었나 → 가장 큰 의사결정 → 왜 그 결정이 옳았나"의 3박자.

예시(결제 모듈 Saga 선택 케이스):

```markdown
#### 한국어 버전
학원 결제 정산 모듈을 설계하면서 2단계 커밋 대신 Saga 패턴을 선택했습니다.
금전 정합성은 유지하되 외부 PG 장애로 전체가 잠기는 위험을 제거했고,
결과적으로 월 결제 10만 건에서 중복 결제 0건과 p99 1.2초를 달성했습니다.

#### 中文版本
在设计补习班的支付结算模块时，我放弃了两阶段提交，选用了 Saga 模式。
这样既保证了资金一致性，又避免了外部 PG 故障让整个系统锁死的风险，
最终在月 10 万笔支付中实现了零重复扣款与 p99 1.2 秒的响应。
```

## 회고 (Keep / Problem / Try / Revisit / Knowledge Gap)

이번 설계·감사 과정에서 **무엇이 잘 됐고, 무엇을 다음 비슷한 프로젝트에서 바꿀지**를 구조화한다. 미래의 나(또는 다음 Chloe)가 같은 실수를 반복하지 않도록 하는 **설계 자체에 대한 ADR**이다.

```markdown
## [모듈명] 설계 회고

### 🟢 Keep — 다음에도 유지할 것
- {구조·절차·판단 중 효과가 좋았던 것}
- 예: "MVP(방안 A) vs 견고한 설계(방안 B) 2안 비교 → 결정 사유를 수치로 정량화한 것"
- 예: "audit에서 Integration Risk 도메인을 일찍 돌려 API 계약 변경을 미리 포착한 것"

### 🔴 Problem — 다음엔 피하거나 바꿀 것
- {이번에 겪은 불편·낭비·놓친 부분}
- 예: "decompose의 결합 관계 분석을 가볍게 하고 넘어가서, audit에서 순환 의존이 뒤늦게 발견됨"
- 예: "의존성 버전 호환성 확인이 리뷰 막판에 추가돼서 일정 지연"

### 🔵 Try — 다음 프로젝트에서 시도할 실험
- {구체적 액션, 가급적 검증 기준까지}
- 예: "decompose 단계에서 Mermaid 뿐 아니라 시퀀스 다이어그램도 추가해서 시간축 의존성 포착"
- 예: "adr 작성 직후 코드베이스에서 `grep 'ADR-NNNN'` 실행 가능한지 dry-run 해보기"
- 예: "결정 사유에 '이 결정이 틀렸다면 어떻게 알게 될지' 한 줄 추가"

### ⏰ Revisit — 이 결정을 다시 볼 조건
- {언제·어떤 신호가 오면 이 ADR/설계를 재검토할지}
- 예: "월 결제 건수 100만 돌파 시 Saga를 별도 서비스로 분리 여부 재검토"
- 예: "외부 PG를 2개 이상 붙이게 되면 통합 계약 레이어 도입 검토"

### 📚 Knowledge Gap — 다음까지 학습할 것
- {이번에 부족했던 지식·스킬·도구}
- 예: "이벤트 소싱 운영 경험 부족 → 다음 프로젝트 전에 최소 PoC 1회"
- 예: "Grafana 대시보드 설계 부족 → 관측성 패턴 공부"
```

**작성 타이밍**: audit가 끝나고 portfolio 커리어 자산 작성 직후. Chloe가 아직 프로젝트 맥락이 뜨거울 때 회고해야 가장 구체적이다.

**회고 장기 누적**: 회고는 한 프로젝트로 끝나지 않는다. 다음 프로젝트 시작 시 **이전 회고의 Try·Revisit·Knowledge Gap을 먼저 읽고** decompose에 진입한다. 이것이 architect-advisor의 복리 효과(compound learning)다.
